# OneCode Desktop — 产品需求文档 (PRD)

> 版本: v1.0 | 日期: 2026-06-19 | 作者: PM | 状态: Draft

---

## 1. 产品概述

### 1.1 背景

OneCode v2 目前以 Web 单页应用形态运行在 Docker 容器内，通过浏览器访问。用户在单个浏览器标签中使用 Claude Code 终端，多任务场景只能靠开多个浏览器标签或反复重启会话。

**现有痛点：**

| 痛点 | 场景 | 影响 |
|------|------|------|
| 多任务靠切标签 | 同时做 bug 修复 + 功能开发 | 上下文频繁切换，容易丢失现场 |
| 浏览器标签不持久 | 关标签即断连接 | 终端会话丢失，需要重新启动 |
| 无法后台常驻 | 等待长时间构建/安装 | 必须保持浏览器打开 |
| 浏览器沙箱限制 | 无法直接访问本地文件系统 | 依赖 gateway 代理，增加延迟 |
| 多终端靠手动 | 每个终端独立开标签 | 无统一管理，无法一键操作 |

### 1.2 目标

**一句话描述：** OneCode Desktop 是一个原生桌面客户端，让开发者在一个窗口内管理多个 AI 终端，一键创建、切换和复用会话。

**季度目标 (2026 Q3)：**

- 发布 OneCode Desktop v0.1，实现多终端管理核心体验
- 内部 dogfooding 替代 Web 版日常开发
- macOS + Linux 双平台可用

### 1.3 产品定位

```
        低控制力                     高控制力
           ┌──────────────────────────┐
  Web 版  │  浏览器标签 × N           │  ← 现状
          │  (零安装，多标签混乱)      │
           ├──────────────────────────┤
 Desktop  │  OneCode Desktop          │  ← 本 PRD
          │  (原生安装，统一管理)      │
           ├──────────────────────────┤
   终端   │  tmux + claude CLI        │  ← 不做
          │  (纯终端，无 GUI)         │
           └──────────────────────────┘
        低效率                         高效率
```

**不替代：** Web 版仍保留（零安装体验），Desktop 是专业用户的高级选项。

| 痛点 | 多终端管理混乱，上下文切换成本高 |

|------|------|
| 角色 | 团队后端/前端工程师 |
| 日常 | 本地开发 + 远程服务器调试 |
| 痛点 | 本地终端 + 远程终端分散管理 |


## 4. 用户故事

### P0 用户故事

**US-1: 首次启动**
> 作为开发者，我打开 OneCode Desktop，看到一个默认的 Claude Code 终端已经在运行，这样我可以立即开始工作。

**US-2: 新建终端**
> 作为开发者，我点击 Tab 栏的 "+" 按钮（或按 Cmd+T），1 秒内出现一个新的 Claude Code 终端，这样我可以并行处理另一个任务。

**US-3: 切换终端**
> 作为开发者，我点击另一个 Tab（或按 Cmd+2），终端内容立即切换，之前终端的输出仍在后台累积，这样我的上下文不丢失。

**US-4: 关闭终端**
> 作为开发者，我点击 Tab 上的 "×" 按钮（或按 Cmd+W），指定终端被关闭并清理，视图自动切换到相邻终端。

**US-5: 重启崩溃终端**
> 作为开发者，我看到某个终端 Tab 变灰（退出状态），点击它后出现"重启"按钮，一键恢复原配置重新启动。

### P1 用户故事

**US-6: 关闭窗口不丢失**
> 作为开发者，我关闭 Desktop 窗口，应用缩小到系统托盘，所有终端继续运行，我点击托盘图标可恢复窗口。

**US-7: 隔天恢复**
> 作为开发者，我第二天打开 Desktop，昨天的 3 个终端自动恢复（相同的命令和工作目录），我不需要重新配置。

**US-8: 快速识别终端**
> 作为开发者，我通过 Tab 栏的颜色指示器快速识别哪些终端还在运行（绿点）、哪些已退出（灰点）。

## 5. 功能规格

### 5.1 终端创建 (P0-2)

**触发方式：**
- 点击 Tab 栏 "+" 按钮
- 快捷键 Cmd/Ctrl + T
- 托盘菜单 → "新建终端"

**创建流程：**
```
点击 [+]
  → Rust spawn PTY (默认: claude --permission-mode bypassPermissions)
  → 返回 {id, pid}
  → 前端创建 xterm 实例
  → 绑定 IPC 事件监听
  → 自动切换到新终端
  → Tab 栏新增一项
```

**默认配置：**
- 命令：`claude`
- 参数：`["--permission-mode", "bypassPermissions"]`
- 工作目录：用户 home 目录 或 上次打开的目录
- 环境变量：继承系统环境 + `HOME`, `TERM=xterm-256color`

**创建时机：** < 1 秒（不含 Claude Code 启动时间）

**最大数量：** 10 个并发终端

**边界条件：**
- 达到 10 个上限 → "+" 按钮置灰，tooltip 显示"已达最大终端数"
- Claude Code CLI 未安装 → 终端显示安装提示（同 Web 版逻辑）
- PTY spawn 失败 → 弹出错误提示，Tab 不创建

### 5.2 Tab 切换 (P0-3)

**交互规格：**
- 点击 Tab → 立即切换，无动画延迟
- 活跃 Tab 有底部绿色下划线
- 切换时自动 replay 非活跃期间的后台输出
- 切换后自动 fit 终端大小

**Tab 栏布局：**
```
┌──────────────────────────────────────────────────┐
│ [main●] [bugfix-42●] [test-run○] [refactor●] [+] │
└──────────────────────────────────────────────────┘
  ● = 绿色(运行中)  ○ = 灰色(已退出)  × = hover显示关闭
```

**Tab 内容：**
- 标签名（可重命名，默认递增编号 term-1, term-2...）
- 状态指示器（颜色圆点）
- 关闭按钮（hover 时显示）

**快捷键：**
| 快捷键 | 功能 |
|--------|------|
| Cmd/Ctrl + 1~9 | 切换到第 N 个终端 |
| Cmd/Ctrl + Shift + [ | 上一个终端 |
| Cmd/Ctrl + Shift + ] | 下一个终端 |

**边界条件：**
- 切换到已退出终端 → 显示退出状态 + 重启按钮
- 非活跃终端累积大量输出 → replay 后自动滚动到底部

### 5.3 终端关闭 (P0-4)

**触发方式：**
- 点击 Tab "×" 按钮
- 快捷键 Cmd/Ctrl + W

**关闭流程：**
```
触发关闭
  → Rust kill PTY (SIGTERM → 3s → SIGKILL)
  → 前端 dispose xterm 实例
  → 移除 Tab DOM
  → 自动切换到相邻 Tab
  → 无剩余 Tab → 自动创建默认终端
```

**确认机制：** 无确认弹窗（与终端关闭标签行为一致，风险可接受）

**边界条件：**
- kill 超时（3s 后 SIGKILL） → 强制清理
- 最后一个终端关闭 → 自动新建默认终端，确保至少有一个
- 关闭正在运行的终端 → 先 kill 再清理

### 5.4 终端重启 (P0-5)

**触发方式：**
- 已退出终端的 Tab → 点击"重启"按钮
- 右键菜单 → "重启终端"
- 命令面板 → "重启当前终端"

**重启行为：**
- 复用原 cmd/args/cwd/env
- 清空终端内容 + ring buffer
- 重新 spawn PTY
- Tab 状态恢复为"运行中"

**自动重启：**
- PTY 崩溃退出 → 自动重启（同 Web 版 pty.js 逻辑）
- 重启策略：指数退避，首次 500ms，最大 30s
- 连续重启超过 10 次 → 停止自动重启，Tab 显示"崩溃"
- 手动重启 → 重置计数器

### 5.5 系统托盘 (P1-1)

**行为：**
- 关闭窗口 → 隐藏到托盘（不退出应用）
- 托盘图标：OneCode 绿色 logo
- 托盘菜单：
  - 新建终端
  - 显示窗口
  - 分隔线
  - 退出 OneCode

**退出：**
- 托盘菜单"退出" → kill 所有 PTY → 保存会话 → 退出进程
- Cmd+Q (macOS) → 同"退出"
- 退出前无确认弹窗（终端内容不保存，仅保存配置）

### 5.6 会话持久化 (P1-2)

**保存内容：**
- 每个 Tab 的元数据：{id, label, cmd, args, cwd, env}
- 不保存终端输出内容（与 tmux 不同，我们只保存配置）

**保存时机：**
- 终端创建/关闭/重命名时 → 实时写入 SQLite
- 应用退出时 → 最终保存

**恢复行为：**
- 启动时读取 SQLite → 按保存的配置重新 spawn PTY
- 恢复顺序：按创建时间排序
- 恢复后 Tab 栏和退出前一致

**不保存的内容：**
- 终端输出历史（PTY 无法真正恢复终端状态）
- 活跃 Tab 选择（恢复后默认激活第一个）
- Preview 面板状态

## 6. 非功能需求

### 6.1 性能

| 指标 | 目标 | 测量方式 |
|------|------|---------|
| 启动时间 | < 3 秒（冷启动到终端可用） | 从双击图标到看到 Claude Code 提示 |
| 新建终端 | < 1 秒（不含 Claude Code 启动） | 点击 [+] 到 xterm 出现 |
| Tab 切换 | < 100ms | 点击 Tab 到内容渲染完成 |
| 内存占用 | 5 终端 < 500MB | Activity Monitor / htop |
| 安装包大小 | < 10MB | .dmg / .AppImage 文件大小 |
| CPU 空闲 | < 5% | 5 终端空闲时 |

### 6.2 安全

| 要求 | 实现 |
|------|------|
| PTY 隔离 | 每个终端独立进程组，互不影响 |
| 权限最小化 | Tauri IPC 白名单，前端仅能调用声明的命令 |
| 无远程代码执行 | 不加载远程 JS/CSS，所有资源本地 |
| 环境变量安全 | 不在 UI 中显示 API_KEY 等敏感变量 |

### 6.3 兼容性

| 平台 | 版本 | 优先级 |
|------|------|--------|
| macOS | 13+ (Ventura) | P0 |
| Linux (x86_64) | Ubuntu 22.04+ / Fedora 38+ | P0 |
| Linux (ARM64) | — | P2 |
| Windows | — | 不做（一人公司资源有限） |

### 6.4 可靠性

| 场景 | 行为 |
|------|------|
| 单个 PTY 崩溃 | 不影响其他终端，自动重启 |
| 所有 PTY 崩溃 | 显示空状态 + "新建终端"引导 |
| 网络断开 | 本地模式无影响，远程模式显示断连提示 |
| 应用更新 | 不自动更新，手动下载新版本 |

| 风险 | 概率 | 影响 | 缓解 |

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| PTY | 伪终端（Pseudo-Terminal），由 portable-pty 创建 |
| Tab | 终端标签页，一个 Tab 对应一个 PTY 实例 |
| Slot | Rust 侧的终端管理单元，包含 PTY + RingBuffer + 元数据 |
| Ring Buffer | 环形缓冲区，每个 Slot 10MB，存储终端输出用于 replay |
| Replay | Tab 切换时，将非活跃期间累积的输出发送给 xterm |
| 托盘 | 系统通知区域的图标，macOS 菜单栏 / Linux 系统托盘 |
| Dogfooding | 团队内部使用自己开发的产品 |
